# Weight and Sleep Read Models

## Outcome

Build fixture-fed, normalized weight and nightly sleep read models plus compact
Hector reads that preserve source, revision, timezone, capability, and privacy
semantics. Imported health summaries must work even when the user has no active
fitness commitment.

## Scope

In scope:

- Consume and preserve M1's `docs/health/withings-provider-contract.md`, fixture
  catalog, source records, cursor semantics, and fake provider.
- Normalize Withings grouped measurements, including exponent decoding, into
  one canonical metric row per measurement. Weight in kilograms is the only
  first-release user-facing body metric; preserve supported optional metrics
  without inventing absent values.
- Normalize sleep summary sessions by provider session ID with start/end,
  timezone, wake-date local sleep date, duration aggregates, stages when
  supplied, score/vitals when supplied, source device, `modified`, `completed`,
  and tombstone state.
- Replace affected normalized values and recompute summaries when a source
  revision advances or completion changes. Handle naps, split/overlapping sleep,
  cross-midnight sessions, timezone changes, and DST deterministically.
- Implement scoped query services and agent read tools for connection freshness,
  latest/7-day/30-day weight trend, and nightly/rolling sleep summaries.
- Correct Hector hot context so health summaries render without an active
  commitment and numeric values/units are not discarded. Include compact
  derived trends only, never raw payloads or stage timelines.
- Add source attribution, privacy, correction/deletion, and isolation tests.

Out of scope:

- Workout normalization or adherence projection, body-composition coaching,
  sleep commitments, stage charts, advanced recovery scoring, medical/apnea
  interpretation, dashboards, partner sharing, and proactive nudges.

## Locked Decisions

- Canonical internal weight unit is kilograms; conversion is a rendering concern.
- Missing/unsupported provider fields remain null. Capability is inferred from
  returned data/model metadata, never assumed from marketing names.
- The user's timezone plus session wake date defines `local_sleep_date`.
- A Withings night is mutable. Provider `modified` and `completed` govern
  replacement/recomputation; revisions are not new nights.
- MVP user reads use aggregate sleep sessions. Stage intervals may remain behind
  the adapter and tests without a user-facing timeline.
- Default LLM context receives latest and rolling summaries only. Raw provider
  data, detailed stages, and health time series stay outside the prompt.
- Existing partner-sharing consent does not authorize health values or derived
  summaries.

## Open Questions to Resolve Autonomously

- Exact deterministic handling of split sleep/naps and overlapping revisions.
- Query-time versus maintained rolling aggregates, clipping/budget rules for hot
  context, and a stable null/unknown representation.
- Whether source-supported stage intervals need persistence now for correct
  revisions or can remain a tested adapter capability until a later UI.

Record decisions and golden examples in the handoff contract.

## Constraints

- All paths remain offline-testable with the M1 fake provider.
- Preserve provider identity, source device, timezone/offset, and revision data.
- Enforce strict user/bot/topic scope on reads and never leak health values to
  logs, generic audit arguments, other users, other topics, or partner context.
- Avoid prompt flooding and do not let absent commitments hide health summaries.
- Corrections and tombstones must propagate to latest/rolling queries.
- Preserve existing Hector tool and adherence behavior.

## Done Criteria

- Fixtures and tests cover measurement exponent/unit conversion, grouped scale
  readings, missing metrics, source attribution, corrections, and tombstones.
- Sleep tests cover cross-midnight sessions, naps, split sleep, DST, missing
  stages, external nights, incomplete-to-complete revisions, overlapping pulls,
  and deletion.
- Latest/7-day/30-day weight outputs and nightly/7-day sleep outputs match
  committed golden results exactly.
- Agent read tools and hot context work with and without active commitments,
  retain values/units, respect budgets, and pass cross-user/topic privacy tests.
- The existing Hector regression suite and all new health tests pass with flags
  off by default.
- Commit `docs/health/weight-sleep-read-model-contract.md` documenting fields,
  units, nulls, local-date, revision, trend, and privacy semantics for M3/M4.

## Touchpoints

M1 health schema/services, `app/services/hot_context_solo.py`, agent tool schemas
and read registry, Hector prompts, tests, and health contract documentation.

## Anti-Scope

Do not project weight or sleep into workout events, create commitments, add a
dashboard, interpret medical signals, or activate provider flags.
