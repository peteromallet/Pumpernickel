# Secure Provider Foundation

## Outcome

Deliver a disabled-by-default, provider-neutral health-ingestion boundary and a
production-shaped Withings adapter contract. OAuth, token rotation, webhook
hints, cursor synchronization, reconciliation, disconnect, and deletion must be
fully executable through frozen fixtures and a fake transport without live
Withings credentials.

## Scope

In scope:

- Add forward migrations for `health_connections`, `health_source_records`,
  `health_sync_runs`, `health_webhook_receipts`, and the minimal durable queue or
  dirty-category state required by the sync worker.
- Define the stable normalized table contracts needed by later milestones for
  body measurements, workouts, sleep sessions, and source-to-event projections;
  later milestones populate and expose them.
- Apply foreign keys, indexes, strict user scoping, FORCE RLS, deny-anon, and
  explicit deletion behavior.
- Add Withings configuration and category flags, all defaulting off. Enabling
  the integration must fail closed when `DATA_ENCRYPTION_KEY` or required
  provider credentials are absent.
- Implement user-bound, signed, expiring, one-time OAuth state; exact redirect
  allowlisting; code exchange; encrypted access/refresh tokens; serialized
  rotating-token refresh; reconnect and revocation states.
- Create `app/services/health_sync/` with a provider interface, Withings HTTP
  adapter, fake transport, normalization boundary, bounded retry/rate-limit
  behavior, pagination, overlap windows, transaction-safe cursor advancement,
  update/tombstone handling, and periodic reconciliation.
- Add authenticated connect/callback/status/resync/disconnect/delete routes and
  a public callback endpoint hardened according to the actual Withings
  notification contract. Notifications only mark a category dirty and return
  quickly; authoritative data is always fetched afterward.
- Preserve the deployed registration paths exactly:
  `/api/health/devices/withings/oauth/callback` and
  `/api/health/devices/withings/notifications`. Both `HEAD` handlers must
  return exactly `200 OK` for the production Partner portal. Replace the
  existing fail-closed OAuth `GET` and notification `POST` stubs only when
  signed one-time state validation, token exchange, and durable notification
  queueing are implemented and tested.
- Add metadata-only operator diagnostics and frozen sanitized/synthetic fixtures
  covering measurements, workouts, sleep, revisions, deletion, pagination,
  token rotation, and errors.

Out of scope:

- User-visible normalized weight, sleep, or workout reads.
- Hector tools, hot-context summaries, adherence projections, charts, partner
  sharing, proactive messages, live provider authorization, vendor approval, or
  production deployment.

## Locked Decisions

- Provider means Withings Public Health Data API; this is cloud sync, not BLE.
- The copied `decisions/withings-health-integration-plan.md` is the product and
  provider starting contract. Record any necessary correction in the milestone
  handoff rather than silently diverging.
- Request a 30-day initial backfill once live credentials exist.
- Webhooks are untrusted change hints; source-record upsert is the authoritative
  idempotency boundary.
- Uniqueness is `(connection_id, resource_type, external_id)`. Where an API
  resource lacks a stable ID, document and test a deterministic provider key.
- Cursors derive from provider modification time, use an overlap window, and
  advance only after the full page set commits.
- Raw provider payload persistence is off by default. Contract fixtures, not
  production payload logs, supply replay/debug coverage.
- Never invoke TurnContext-bound LLM tool handlers from background sync code.
- Disconnect and delete are distinct, explicit, authenticated operations.

## Open Questions to Resolve Autonomously

- Whether the existing scheduled-job substrate can safely host non-user-facing
  reconciliation or a small dedicated worker/queue is cleaner.
- The cursor JSON shape, deterministic fallback external keys, sanitized error
  taxonomy, and exact notification validation supported by Withings.
- The smallest provider-neutral interface that supports Withings without
  speculative multi-provider abstraction.

Document decisions and tests. Device availability, API-pack entitlement,
production promotion, retention policy, and real credentials are operator
prerequisites, not blockers for this milestone.

## Constraints

- Tests must be deterministic, offline, and credential-free.
- Preserve existing manual events, commitments, transport authentication, and
  scheduler behavior.
- Never print or persist unencrypted tokens, OAuth codes, raw measurements, or
  full payloads in logs/audit text.
- Token refresh must remain correct under concurrent workers and provider token
  rotation.
- Bound request size, timeouts, retry counts, backoff, and rate-limit behavior.
- Follow the repository's forward-only migration and validation conventions.
- Do not touch unrelated user-owned changes or broaden auth defaults.

## Done Criteria

- Migration/static/DB tests prove schema constraints, indexes, RLS, deletion,
  and downgrade documentation.
- OAuth tests cover state replay, wrong user, expiry, redirect mismatch, failed
  exchange, revocation, and concurrent refresh-token rotation.
- Sync tests cover duplicate/reordered/missing notifications, pagination,
  provider revisions/deletions, cursor crash atomicity, rate limits, and missed
  notification reconciliation.
- A fake-provider end-to-end test completes connect → 30-day fixture backfill →
  webhook hint → changed-record fetch → reconcile → disconnect/delete with zero
  duplicate source records.
- Startup and the existing suite pass with all health flags off and no provider
  secrets.
- Commit `docs/health/withings-provider-contract.md`, a fixture catalog, and the
  fake-provider harness as the explicit handoff to M2.

## Touchpoints

## Anti-Scope

Do not add a fitness dashboard, infer health advice, activate flags, require live
credentials, refactor generic transports, or write provider data directly into
`mediator.events`.
