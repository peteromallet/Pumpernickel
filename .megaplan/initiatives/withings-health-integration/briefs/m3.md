# Workout Adherence Projection

## Outcome

Normalize imported workouts as independent evidence, then add a versioned,
reversible, exact-once projection that can mark one compatible existing workout
commitment done without creating commitments, guessing through ambiguity,
double-counting, or deleting manual testimony.

## Scope

In scope:

- Consume the M1 provider contract and M2 read-model contract without weakening
  their source, revision, privacy, and cursor invariants.
- Normalize workout sessions with provider ID, source device, start/end,
  timezone/local date, type taxonomy, duration, distance, steps, energy,
  available heart-rate summaries, revision, and tombstone state.
- Add scoped recent-workout queries/tools and compact weekly workout context.
- Identify and consolidate onto the production adherence implementation; retire
  or explicitly redirect the duplicate implementation/test path.
- Define explicit mapping plus a conservative deterministic matcher. Automatic
  projection is permitted only when exactly one active Hector fitness
  commitment is type-compatible and has an eligible local-date slot. Everything
  else remains visible as unmatched or needs confirmation.
- Implement a projection ledger with source ID, event ID, commitment ID,
  projection version/status, decision reason, and timestamps. Enforce at most
  one active projection per source workout.
- Create, supersede, or reverse only projection-owned `mediator.events` when a
  source changes, is deleted, or is rematched. Preserve manual events and link
  likely manual/device duplicates in the read model so they count once.
- Fix the current unsafe rule under which any linked numeric event may count as
  done; completion must be type-safe for the commitment/event combination.

Out of scope:

- Creating commitments, probabilistic/LLM matching, inferred missed/excused
  events, projecting weight or sleep, broad adherence redesign, dashboards, and
  proactive workout messaging.

## Locked Decisions

- Imported workouts remain canonical health records whether or not they match a
  commitment. Projection is derived and rebuildable.
- Exactly one compatible active commitment plus an eligible local-date slot is
  required for automatic projection. Two candidates means no projection.
- A retry or concurrent worker cannot create two source records, projections, or
  adherence completions.
- Provider revision/deletion may mutate only its own projected event through an
  auditable supersession/reversal path.
- Manual reports are user testimony and are never deleted or overwritten by
  provider data.
- Projection is controlled by a separate feature flag that defaults off.

## Open Questions to Resolve Autonomously

- The normalized Withings-to-Hector workout taxonomy table.
- A deterministic tolerance window and linking representation for likely
  manual/device duplicates.
- Whether projection reversal is best represented by deletion, supersession, or
  an explicit inactive state given existing event contracts; prefer an auditable
  approach and document compatibility implications.
- The safest migration path to one canonical adherence implementation.

Resolve these with a committed decision table and invariant tests.

## Constraints

- Preserve existing commitment creation/update/close and manual `log_event`
  behavior.
- Use the user's timezone and local slot semantics, including DST boundaries.
- Projection updates must be transactional, idempotent, and scoped by user,
  fitness topic, and Hector.
- Unmatched/ambiguous workouts remain queryable and never alter adherence.
- No source revision may mutate unrelated or manual events.

## Done Criteria

- Normalizer fixtures cover representative workout types, missing optional
  metrics, timezone/DST, revisions, deletions, and unknown categories.
- Retries and concurrent runs produce exactly one active projection and one
  adherence completion.
- One-candidate matches project; zero/two-candidate cases remain unprojected;
  wrong metric types cannot satisfy workout commitments.
- Provider revisions/deletions reverse or supersede projection-owned events and
  recompute adherence correctly.
- Manual/device duplicates remain auditable but count once; manual testimony is
  untouched.
- Production and test callers converge on one adherence implementation, and the
  existing focused 108-test Hector baseline plus new regressions pass.
- Commit `docs/health/workout-projection-contract.md` with taxonomy, matching,
  duplicate, reversal, and type-safety decision tables for M4.

## Touchpoints

M1/M2 health services and schema, `mediator.events`, tool schemas/registry,
`app/services/adherence.py`, the duplicate commitments calculator/tests, Hector
hot context/prompts, and projection contract documentation.

## Anti-Scope

Do not enable projection by default, guess ambiguous matches, create commitments,
infer failure, delete manual reports, or redesign unrelated fitness behavior.
