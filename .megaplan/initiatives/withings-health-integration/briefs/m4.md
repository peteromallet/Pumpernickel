# Product Hardening and Rollout Readiness

## Outcome

Deliver an operator-ready, privacy-reviewed, observable integration package that
can be enabled later by a human with Withings credentials. Prove the complete
system through an offline simulated canary and leave explicit evidence and a
runbook for live authorization, deployment, staged enablement, rollback, and
dogfood validation.

## Scope

In scope:

- Consume and verify all three upstream handoff contracts and invariant suites.
- Finish authenticated connection/status/resync/disconnect/delete/export user
  flows and metadata-only operator diagnostics.
- Add sanitized health/sync metrics, freshness and error classifications, alert
  conditions, and failure drills for stale cursors, reauthorization, rate limits,
  webhook-without-fetch, duplicates, revisions, and projection drift.
- Add a feature-flag/canary matrix with enable order: connection → weight/sleep →
  workouts → projection. Every flag remains off by default.
- Add explicit export, retention-choice, provider unsubscribe/revoke, local
  delete, and cross-user isolation behavior.
- Add an optional fixture-driven weekly summary behind a separate default-off
  flag only if it fits Hector's existing scheduling/prompt architecture without
  broad refactoring. No proactive message is required for completion.
- Document migration application/verification, configuration, provider app
  setup, live smoke tests, rollback, data deletion, operator checks, known
  capability limitations, and post-launch dogfood gates.
- Run end-to-end synthetic canaries for weigh-in → trend, sleep revision →
  rolling summary, and workout → adherence → reversal.

Out of scope:

- Actual Withings app approval, real OAuth/device authorization, production
  deployment, contacting users, waiting through dogfood, partner sharing, legal
  certification, a dashboard, advanced recovery/body-composition features, or
  medical claims.

## Locked Decisions

- Live rollout is an operator action after this chain. The milestone reports
  release readiness and explicitly records live validation as pending.
- Staleness beyond 24 hours is the default alert threshold for a connected,
  enabled category unless test evidence justifies a stricter category threshold.
- Health values remain absent from default admin listings, logs, tool audit text,
  and partner context; operator pages default to IDs, status, counts, and times.
- Safe rollback begins by disabling projection/category/connection flags in that
  order and then follows documented data-handling steps.
- Dashboard and partner sharing remain deferred.

## Open Questions to Resolve Autonomously

- Exact launch checklist ordering, synthetic canary matrix, metric names, and
  alert thresholds below the 24-hour hard freshness gate.
- Whether a weekly digest is small and safe enough for this milestone; if not,
  document it as a follow-up without weakening readiness.
- The clearest operator evidence report format and proof-map contents.

Retention duration, EU legal review, vendor plan/approval, real cohort size, and
device availability are explicit human prerequisites, not autonomous decisions.

## Constraints

- No live credentials, users, network dependency, deployment, or time-based wait
  is required for completion.
- Exports and destructive operations require authenticated, user-scoped flows.
- All defaults remain inert and backward compatible.
- Tests and operational output must not reveal tokens, raw payloads, or health
  measurements outside explicitly authenticated user reads.
- Do not accept flaky simulations or ignore failing regressions to declare ready.

## Done Criteria

- Offline E2E tests prove weigh-in → trend, late sleep revision → rolling
  summary, workout → exactly-once adherence, and provider deletion → reversal.
- Export/delete tests prove complete scope, isolation, revoke/unsubscribe intent,
  and absence of orphaned projections or normalized records.
- Simulations cover stale sync, lost notification recovery, rate limits,
  reauthorization, duplicate records, cursor crash, and projection drift.
- A repository scan/test proves credentials, raw payloads, and health values do
  not appear in prohibited logs/admin/audit/prompt surfaces.
- All existing and new tests pass with all flags off.
- Commit `docs/health/rollout-runbook.md` and a verification/proof report that
  explicitly distinguishes automated evidence from pending live validation.
- Provide a proof map suitable for a chain completion manifest, covering each
  milestone's contract and invariant tests.

## Touchpoints

Health routes/services/tools, admin diagnostics, metrics/observability,
configuration and `.env.example`, deletion/export services, Hector context and
optional scheduling, migrations/runbooks, and end-to-end tests.

## Anti-Scope

Do not claim a live rollout, add a dashboard, enable flags, share health data,
make medical claims, or broaden platform architecture beyond what release
readiness requires.
